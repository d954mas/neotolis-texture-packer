# Defold demo — three-way atlas comparison

In-repo Defold project (ROADMAP Phase 5) that proves atlases exported by
**ntpacker** load and render correctly in a stock Defold build, side by side
with the two other atlas pipelines Defold supports, over the **same source
images** so the results are directly diffable.

## Which file comes from where

Each example folder (`examples/basic`, `examples/rotate`, `examples/anim_trim`)
holds the same atlas built three ways. The filename suffix says who made it:

| file | pipeline | provenance |
|---|---|---|
| `examples/<name>/<name>.tpinfo` + `.tpatlas` + `<name>-N.png` | **TexturePacker** | upstream originals, **untouched reference** (CodeAndWeb TexturePacker via extension-texturepacker; the `.tps` shows the settings used) |
| `examples/<name>/<name>_d.atlas` | **Defold native** | plain Defold `.atlas` text, packed by bob at build time |
| `examples/<name>/<name>_nt.tpinfo` + `.tpatlas` + `<name>_nt-N.png` | **ntpacker** | generated by our `defold` exporter from `defold-demo.ntpacker_project` |
| `examples/<name>/<name>_nt_json.json` + `<name>_nt_json-N.png` | ntpacker (JSON) | our `json-neotolis` exporter — full-capability pages, kept tracked for reference (not used by the scene) |

`<name>` is `basic`, `rotate`, or `anim_trim`. Suffix key: **`_d` = Defold
native, `_nt` = ntpacker, no suffix = TexturePacker original.**

## What you should see

Open this folder in the **Defold editor (1.12.4)** and run. The boot collection
`/examples/compare/compare.collection` lays out a labelled 3×3 grid:

- **columns** = the three packers, labelled `Defold` · `TexturePacker` · `ntpacker`
- **rows** = the three example atlases, `basic` · `rotate` · `anim_trim`

Each cell shows one representative **static** sprite and, for the `basic` and
`anim_trim` rows, the **flipbook animation playing** (`BoxFlip` at 4 fps,
`anim_trim` at 6 fps). Because every column packs the identical source PNGs, the
regions and the animations should look the same across a row — that visual
equality is the proof that the ntpacker export is correct.

Source images (the inputs every pipeline packs):
`examples/basic/original/`, `examples/rotate/{a,b,c}.png`,
`examples/anim_trim/anims/sq1..sq10.png`.

## Regenerating the ntpacker exports

The `_nt.*` / `_nt_json.*` files are produced by `tp_demo_export`, a small
driver that loads `defold-demo.ntpacker_project`, scans each atlas's sources for
sprites, and runs its `defold` + `json-neotolis` targets to the project's
`out_path`s. One command regenerates everything deterministically:

```sh
cmake --preset native-release
cmake --build --preset native-release --target tp_demo_export
python examples/defold-demo/regen_exports.py
```

Notes on the packing settings (`defold-demo.ntpacker_project`):

- **Page sizes match TexturePacker for a fair comparison.** `basic` and
  `anim_trim` use `max_size` 256 / 128 (the TexturePacker page dimensions).
- **`rotate` uses `max_size` 512, `padding` 0.** TexturePacker fits rotate into a
  single 256×256 page only by rotating sprites 90°; the ntpacker **Defold** export
  is **identity-only** (no rotation until engine #285), so the 256-tall `a` sprite
  cannot share a 256 page with the others. Identity packing therefore
  legitimately needs more/larger pages than TexturePacker — that is expected. The
  `json-neotolis` target keeps full capability (it rotates) and packs tighter.
- Consequently the ntpacker Defold `basic` atlas is **3 pages** (a paged atlas),
  so its sprites in the scene use `sprite_paged_atlas.material` — matching
  TexturePacker's `basic` (2 pages, also paged). `rotate` and `anim_trim` are
  single-page (`sprite.material`).

### Known shim (temporary)

`regen_exports.py` rewrites each `_nt.tpatlas` `file:` field from the exporter's
relative basename (`basic_nt.tpinfo`) to a project-absolute resource path
(`/examples/basic/basic_nt.tpinfo`). extension-texturepacker declares that field
as a Defold resource, resolved from the project root — a bare basename resolves
to `/basic_nt.tpinfo` (missing) and the build fails. The proper fix is in the
`tp_export_defold` exporter (emit the absolute path); this shim covers it until
then. Page-image names **inside** the `.tpinfo` stay bare on purpose (those are
correctly resolved next to the `.tpinfo`).

## Provenance & license

`examples/`, `input/`, and the TexturePacker comparison assets are copied from
[defold/extension-texturepacker](https://github.com/defold/extension-texturepacker)
(MIT, see `UPSTREAM-LICENSE`), as of tag **2.7.0**. The `_d.atlas`, `_nt.*`, and
`examples/compare/` files are added by this project. `game.project` was rewritten
to consume the extension as a **pinned dependency** instead of in-tree source,
and its bootstrap now points at the comparison collection. The upstream
per-example collections (`basic`/`rotate`/`anim_trim`/`skins`) remain and can be
booted via `[bootstrap] main_collection` if you want to inspect one pipeline.

## Version pinning (lock-step rule)

The extension is version-locked to Defold releases. This demo pins:

- extension-texturepacker **2.7.0** ↔ Defold/bob **1.12.4**
- dirtylarry at commit `6f2070e909a9`

Bumping either pin is a deliberate small PR that changes the dependency URL and
the bob.jar version together (see `docs/research/defold.md`).

## Building headless (CI proof — follow-up)

```sh
cd examples/defold-demo
java -jar bob.jar resolve                       # fetch pinned dependencies
java -jar bob.jar --archive build --variant headless
```

A clean bob build compiles every atlas + the comparison collection and validates
that every animation/frame id the scene references exists — so a green build is
the real proof. **bob for Defold 1.12.4 requires JDK 25**; if your JDK is older,
this step is a follow-up. Until then, `python` structural validation (page PNGs
exist and are co-located; every id referenced by the scene exists in its target
atlas) stands in.
