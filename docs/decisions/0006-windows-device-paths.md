# 0006 — Windows device/verbatim пути (`\\?\`, `\\.\`) в project identity

**Дата:** 2026-07-12
**Статус:** accepted
**Принял:** lead (делегированные полномочия)
**Реализовано в:** `tp_path_canonical_lexical` (master spec §5.1, §16)

## Решение

При канонизации Windows-пути (после переписывания `\` → `/`) префикс
`\\?\` (verbatim namespace) трактуется как **прозрачный лексический алиас**:

- `\\?\X:\...` (`//?/X:...`) — префикс срезается, остаток канонизируется как
  drive-путь. `\\?\C:\work\p` ≡ `C:\work\p`.
- `\\?\UNC\server\share\...` (`//?/UNC/...`, компонент `UNC` сравнивается
  ASCII-регистронезависимо) — переписывается в `//server/share/...` и
  канонизируется как UNC.

Любая другая форма `\\?\...` и **все** `\\.\...` (device namespace) —
структурная ошибка с токеном `path_device` (`TP_STATUS_PATH_DEVICE`).

POSIX-хост не затронут.

## Контекст

Master spec §5.1: у одного файла — одна идентичность (канонический путь).
`\\?\C:\work\p` и `C:\work\p` адресуют **один и тот же** файл; без этого
правила они получили бы разные identity (наивно `//?/C:/work/p`), что ломает
дедупликацию проекта и сравнение путей. `\\?\` — чисто лексический
verbatim-маркер (обходит нормализацию Win32), поэтому его срезание не меняет
целевой файл: это безопасный алиас для drive- и UNC-форм.

Device namespace `\\.\` (и не-drive/не-UNC формы `\\?\`, напр.
`\\?\GLOBALROOT\...`, `\\?\Volume{GUID}\...`) адресует устройства/тома, а не
файлы проекта. Такой путь никогда не является `.ntpacker_project`, поэтому
принимать его — значит молча породить нефайловую identity. Отклоняем явно
структурной ошибкой (не abort — AGENTS.md invariant).

Правило чисто лексическое; realpath/симлинки слой F1-00 добавляет поверх.
Production-goldens для обеих host-моделей находятся в
`packer/tests/test_identity_path.c`; контракт задают master spec §5.1 и это решение.
